Why this is the right solution

The corpus is 14% Devanagari characters but 33% Devanagari bytes (each Hindi char = 3 UTF-8 bytes). Three independent levers, in order of impact:

  • Training dynamics — the baseline was badly under-optimized. AdamW + warmup→cosine LR + grad-clip + weight tying + scaled init did the most (−0.36) with no extra params.
  • Tokenizer — a corpus-trained byte-level BPE (vocab 2048, ~2.35 B/tok) halves sequence length, so each fixed step sees ~2× more text and the window spans ~2× more characters. Lossless (decode(encode(x))==x) with raw-byte fallback for any unseen UTF-8.
  • Modern components — RoPE (Su 2021), RMSNorm (Zhang 2019), SwiGLU (Shazeer 2020) lowered bpb and freed params (RoPE has no learned table), paying for a 5th layer; Muon (Jordan 2024) — orthogonalised momentum updates — beat AdamW under the hard step cap; and QK-Norm stopped attention saturating so Muon could push harder.

Because bpb is measured per byte, the tokenizer gain is genuine compression. Run 4 is an honest negative result: buying context by halving the batch increased gradient noise over the fixed step budget and lost.
